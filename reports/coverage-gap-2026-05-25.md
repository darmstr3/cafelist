## Method
For each (city, neighborhood) extracted from parsed intent we compute:
- `query_count` — demand signal.
- `avg_quality_score` — evaluator output, 0–10.
- `thin_result_rate` — fraction of runs returning fewer than 3 picks.
- `dominant_failure_mode` — most common non-`ok` failure mode.

```
priority_score = log10(1 + queryCount)
               * max(0, 10 - avgQuality)
               * (1 + thinResultRate)
               * 100    // weight vs hardcoded seeds (seeds cluster 80–100)
```

Missing `avgQuality` is treated as max gap (10) — a run with no
evaluator output is itself a strong signal something's off.

  `npm run` too).** The script targets `ON CONFLICT (city,

  idx_scout_priority_city_hood`. This is a schema/script mismatch,
  not transient — every coverage_gap row whose (city, neighborhood)
  collides with an existing seed will fail the same way. The script
  catches the error and continues to write this report; the
  `scout_priority` table is unchanged from the previous run.
